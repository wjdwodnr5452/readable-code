### 강의구성
1. 추상
2. 논리, 사고의 흐름
3. 객체 지향 패러다임
4. 객체 지향 적용하기
5. 코드 다듬기
6. 리팩토링 연습
7. 기억하면 좋은 조언들

### 클린코드, 리팩토링 과정의 베스트 프랙티스는 테스트 코드가 생성되는 사이클이다.
-> 리팩토링 대상/범위확인 - 기능 보장을 위한 테스트 코드 작성 - 리팩토링 - 테스트 코드로 검증

롬복은 사용하지 않음

# 챕터2
### 코드가 잘 읽힌다.
- 이해가 잘된다.
- 유지보수가 하기가 수월하다.
- 우리의 시간과 자원이 절약된다.

### 잘 읽을 수 있는 코드를 작성하는 건
- 클린코드

### 프로그램의 정의
- 프래그램은 데이터와 코드의 조합
- 데이터는 식재료
- 코드는 식재료를 통해 만드는 요리

### 추상과 구체
- 추상은 뽑을 추 코끼리 상(형상 상)으로 구체적인 정보들에서 내가 어떤 이미지를 뽑아낸다는 뜻
- 사물을 정확하게 이해하기 위해서 사물이 지니고 있는 여러 가지 측면 가운데에서 특정한 측면만을 가려내어 포착하는 것이다.
- 구체에서 추상을 할 때 추상화 레벨을 통해 함
```
사물을 정확하게 이해하기 위해서 사물이 지니고 있는 여러 가지 측면 가운데에서 특정한 측면만을 가려내어 포착하는 것이다. 
어떤 일면만을 추상하는 것은 다른 측면을 버린다는 것과 같다.
```


### 고수준 : 추상화 레벨이 높다. 인간이 읽을 수 있을 수록
### 저수준 : 추상화 레벨이 낮다. 기계어 가까울 수록

### 추상으로부터 구체를 유추하지 못한 이유
1. 추상화 과정에서 중요한 정보를 부각시키지 못했다.
- 상대적으로 덜 중요한 정보를 남기고 중요한 정보는 제거 했다.
2. 해석자가 동일하게 공유하는 문맥이 없다.
- 중요한 정보의 기준이 다를 수 있다.
- 즉 도메인 영역 별 추상화 기준이 다를 수 있다.

구체로부터 추상을 해내고 정보를 덜어내고 가장 중요한 정보는 남기고 그 추상화 과정은 컴퓨터 과확에 본질이다.
