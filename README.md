# 챕터2

## 추상 

### 코드가 잘 읽힌다.
- 이해가 잘된다.
- 유지보수가 하기가 수월하다.
- 우리의 시간과 자원이 절약된다.

### 잘 읽을 수 있는 코드를 작성하는 건
- 클린코드

### 프로그램의 정의
- 프래그램은 데이터와 코드의 조합
- 데이터는 식재료
- 코드는 식재료를 통해 만드는 요리

### 추상과 구체
- 추상은 뽑을 추 코끼리 상(형상 상)으로 구체적인 정보들에서 내가 어떤 이미지를 뽑아낸다는 뜻
- 사물을 정확하게 이해하기 위해서 사물이 지니고 있는 여러 가지 측면 가운데에서 특정한 측면만을 가려내어 포착하는 것이다.
- 구체에서 추상을 할 때 추상화 레벨을 통해 함
```
사물을 정확하게 이해하기 위해서 사물이 지니고 있는 여러 가지 측면 가운데에서 특정한 측면만을 가려내어 포착하는 것이다. 
어떤 일면만을 추상하는 것은 다른 측면을 버린다는 것과 같다.
```


### 고수준 : 추상화 레벨이 높다. 인간이 읽을 수 있을 수록
### 저수준 : 추상화 레벨이 낮다. 기계어 가까울 수록

### 추상으로부터 구체를 유추하지 못한 이유
1. 추상화 과정에서 중요한 정보를 부각시키지 못했다.
- 상대적으로 덜 중요한 정보를 남기고 중요한 정보는 제거 했다.
2. 해석자가 동일하게 공유하는 문맥이 없다.
- 중요한 정보의 기준이 다를 수 있다.
- 즉 도메인 영역 별 추상화 기준이 다를 수 있다.

구체로부터 추상을 해내고 정보를 덜어내고 가장 중요한 정보는 남기고 그 추상화 과정은 컴퓨터 과확에 본질이다. 


## 이름짓기

### 프로그래머가 가장 힘들어하는일?
- 이름짓기가 제일 힘들다고 한다.

### 이름을 짓는다는 행위는 추상적 사고를 기반으로한다.
- 추상적 사고는 표현하고자 하는 구체에서 정말 중요한 핵심 개념만을 추출하여 잘 드러내는 표현
- 우리 도메인의 문맥 안에서 이해되는 용어

### 이름짓는법 - 단수와 복수를 구분하기
- 끝에 -(e)s를 붙여 어떤 데이터(변수, 클래스 등)가 단수인지, 복수인지를 나타내는 것만으로도 읽는 이에게 중요한 정보를 같이 전달할 수 있다.

### 이름짓는법 - 이름 줄이지 않기
- 줄임말이라는 것은 가독성을 제물로 바쳐 효율성을 얻는 것으로, 대부분 잃는 것에 비해 얻는 것이 적다.
- 즉, 자제하는 것이 좋으나, 관용어처럼 많은 사람들이 자주 사용하는 줄임말 정도는 존재한다.
   - ex) column -> col, latitude -> lat, longitude -> lon  // 많은 사람들이 사용해서 누가 봐도 이 용어다 라고 하는 줄임말 정도만 쓰는 것이 좋다.
   - ex) count -> cnt (추천)
- 줄이는 이름은 자주 사용하는 줄임말이 이해될 수 있는 것은 사실 문맥 때문이다.
  
 ### 이름짓는법 - 은어/방언 사용하지 않기
 - 농담에서 파생된 용어, 일부 팀원/현재의 우리 팀만 아는 용어 금지 
    - 새로운 사람이 팀에 합류했을 때 이 용어를 단번에 이해할 수 있는가?
 - 도메인 용어 사용하기
    - 도메인 용어를 먼저 정의하는 과정(ex. 도메인 용어 사전)이 먼저 필요할 수도 있다.
  
### 이름 짓기 - 좋은 코드를 보고 습득하기
- 비슷한 상황에서 자주 사용하는 단어, 개념 습득하기
  -> ex. pool, candidate, threshold 등
